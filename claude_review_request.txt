Please review the following proposed changes for the current `Icon_Changer` codebase and tell me whether you approve of them.

Important:
- This is a review/approval request only.
- Do not implement changes.
- I want you to evaluate whether these are the right changes, whether there are any edge cases I am missing, and whether you would modify any of the proposals before coding them.

Proposed changes to review:

1. Make the Change Icon UI responsive in smaller windows

Problem:
When the app window is not maximized, the Step 1 area becomes too cramped and it is hard to see which shortcut/icon is being changed.

Desired outcome:
- The shortcut list should remain readable in smaller window sizes.
- Long shortcut names/targets/icon-source text should wrap or truncate intelligently instead of getting clipped.
- The preview / Step 2 area should not take too much space away from the Step 1 batch list.

Proposed changes:
- In `ui/tabs/change_icon_tab.py`:
  - Rework the layout so the Step 1 batch list gets more usable space in smaller windows.
  - Reduce fixed-size pressure from Step 2 and the preview area.
  - If feasible, make the layout adapt when the window gets narrow so Step 2 does not visually crowd Step 1.
  - Reduce unnecessary padding / fixed heights if needed.
- In `ui/components/batch_list_frame.py`:
  - Add a second metadata line per shortcut row that shows the current icon source.
  - Make long text wrap using `wraplength` and left alignment.
  - Ensure rows stay readable in narrow window widths.
  - Update wrapping dynamically when the widget resizes if needed.
- In `ui/components/live_preview.py` or related layout code:
  - Reduce preview width if necessary so the batch list has more room.

Goal:
Even in a smaller window, I should clearly see which shortcuts are queued and what their current icon source is.

2. Add two separate restore options

Problem:
Right now restore behavior is too broad. I want two restore options:
- Restore current session only
- Restore all shortcuts ever modified by the app

Proposed changes:
- In `ui/tabs/utilities_tab.py`:
  - Replace the single restore action with two actions:
    - `Restore Current Session`
    - `Restore All Tracked Shortcuts`
- Update confirmation dialogs accordingly:
  - Current session:
    - “Restore only shortcuts modified since this app launch?”
  - All tracked:
    - “Restore all shortcuts recorded in IconForge history, including previous sessions?”

3. Track current-session modified shortcuts in memory

Desired behavior:
- Keep track of which shortcuts were modified during the current app session.
- This should be separate from the full manifest history.
- Restore Current Session should only operate on those session-tracked shortcuts.
- Restore All Tracked Shortcuts should continue to use the full manifest.

Proposed implementation:
- Store a session-level set/list on the app instance, such as `self.app.session_modified_shortcuts`.
- Initialize it when the app starts.
- Whenever Apply succeeds for a shortcut in `ui/tabs/change_icon_tab.py`, add that shortcut’s manifest key/original path to the session set.
- Use that set to filter entries for Restore Current Session.

4. Be conservative with missing shortcuts during restore

Important requirement:
Do not automatically delete manifest state just because a shortcut is missing.

Proposed behavior:
- If the shortcut file no longer exists:
  - Skip it
  - Report it in the result dialog as missing/skipped
  - Keep the manifest entry intact
- If the shortcut was manually renamed and can still be found through existing tracking, restore should continue safely.
- If the original icon source no longer exists:
  - Restore the original name if possible
  - Report that icon restoration could not be completed
  - Preserve metadata conservatively instead of auto-cleaning it up

Result dialog should clearly separate:
- Restored successfully
- Skipped because shortcut is missing
- Partially restored because original icon source is unavailable
- Failed because of permission / COM / other errors

5. General restore behavior expectations

Please evaluate whether this policy is sound:
- Prefer “skip and preserve metadata” over “assume it is safe to forget”
- Do not silently discard recovery information
- Do not remove manifest entries unless the shortcut has been successfully restored to a known-good final state
